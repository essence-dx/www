# DX WWW Framework Structure & Coding Standards

## Overview

DX WWW is a React-familiar web framework backed by the DX toolchain: a Rust-powered dev server, dx-style CSS generation, source-owned Forge packages, dx-check receipts, and static export paths. The public starter story should be simple enough for React/Next.js developers to understand immediately while keeping DX-native compiler features available for advanced use.

## Core Philosophy

- **React-Familiar Public Surface**: `.tsx` routes and an App Router-shaped `app/` directory are the default developer entry point.
- **CSS-First Styling**: dx-style scans source and generates normal CSS for WWW apps. Binary style output is deprecated for now and should not be part of the public path.
- **File-System Routing**: Convention over configuration
- **Forge Packages**: Official packages are source-owned slices with provenance and receipts instead of template-local dependency sprawl.
- **Checkable Output**: dx-check validates route structure, generated CSS, assets, hardcoded colors, receipts, and template-readiness state.

---

## File Extensions

| Extension | Purpose | Example |
|-----------|---------|---------|
| `.tsx` | Public default route/component source | `app/page.tsx`, `components/icons/icon.tsx` |
| `.html` | Internal preview/export fixture, not public route authoring | `.dx/www/output/index.html`, legacy fixture snapshots |
| `.tsx` | Advanced/internal DX-native component source | `Button.tsx`, `Card.tsx` |
| `.css` | Public stylesheet entry, authored tokens, or generated dx-style output | `styles/globals.css`, `styles/theme.css`, `styles/generated.css` |

Deprecated for now: `.sr` binary style artifacts. Do not generate or document `.sr` as part of the public CSS workflow until the CSS-first path is mature and the binary style lane is re-approved.

## Public WWW Project Contract

The official WWW starter path is TSX-first. Package workers should plug into this shape instead of adding alternate public authoring surfaces:

```text
app/layout.tsx
app/page.tsx
components/auto-imports.ts
components/icons/icon.tsx
styles/globals.css
styles/theme.css
styles/generated.css
.dx/forge/receipts/*
.dx/www/output/*
```

`app/page.tsx` should stay thin: own metadata and render the source-owned template app surface. Package features may materialize into `components/`, `lib/`, `server/`, `styles/`, and Forge/status receipts when a lane is installed, but the minimal starter docs must not claim optional package surfaces as emitted files.

### Folder Boundary

The public authoring model is `app/` only. New routes, route handlers, layouts, loading states, errors, metadata, and user-editable page surfaces belong there, including route handlers under `app/api/**/route.ts`. `pages/` is not part of the WWW starter contract and must not be presented as a supported authoring tree for new apps.

The app imports only `styles/globals.css`. That file is the public stylesheet entry and imports `styles/theme.css` plus `styles/generated.css` before the browser loads the final CSS. `styles/theme.css` is the human-owned token input for `dx style`; developers edit it when they want to change design tokens. `styles/generated.css` is tool-owned evidence/output from `dx style build`; developers should not hand-edit it, and apps should not import it directly from app code.

Generated framework state lives under `.dx/`. Build output belongs in `.dx/www/output` and is never source to edit; serializer machine files belong in `.dx/serializer`; Forge, Check, Style, Icons, and Imports write their receipts in their own `.dx/*` areas. Anything outside `app/`, `components/`, `server/`, `lib/`, `styles/`, `content/`, `openapi/`, `public/`, or `dx` should be treated as package/tool-owned unless explicitly documented.

Static preview fixtures are internal WWW tool output. They are not public authoring surfaces, should not appear in starter instructions, and should only be materialized into generated output such as `.dx/www/output` or temporary verification directories.

### Runtime Layer Names

First-user config should not expose runtime crate plumbing. Architecture contracts can name runtime layers by what they do: `foundation`, `html`, `javascript`, `wasm`, `browser`, `protocol`, and `server`. Implementation crate names such as `dx-www-browser` and `dx-www-browser-micro` stay in protected-crate/provenance lists, not in the starter `dx` file.

## Package Worker Slot Contract

Package worker registry fixtures currently live under `examples/onboard/template-surface-registry.ts`, not the minimal `examples/template` starter. Each surface declares:

- source and materialized files
- the stable `data-dx-*` selectors Zed can target
- package ids and roles
- required env vars and receipt paths
- the allowed exports for that lane
- forbidden behavior, including fake success states and template-local `node_modules`

The shell renders this registry as a real dashboard surface with `data-dx-component="template-worker-registry"`. That gives package workers one professional place to plug in auth, settings, billing, data, query, state, reactive store, i18n, docs, motion, 3D, WASM, AI, automations, icons, and token/money meters.

## Public TSX Authoring

DX-WWW public starter work uses React-shaped TSX and an App Router-shaped `app/` tree. New package workers should add source-owned TSX files, typed helpers, CSS tokens, receipts, and docs in the package lane. They should not introduce a second public component language.

### Route Source

```tsx
export default function Page() {
  return (
    <main className="mx-auto grid max-w-4xl gap-6 p-6">
      <section className="grid gap-3">
        <p className="text-sm font-medium text-neutral-500">DX-WWW</p>
        <h1 className="text-3xl font-semibold text-neutral-950">Source-owned WWW route</h1>
        <p className="text-base text-neutral-700">
          Forge packages install visible files that developers can inspect, edit, and remove.
        </p>
      </section>
    </main>
  );
}
```

### Component Source

```tsx
import type { ReactNode } from "react";

type WwwPanelProps = {
  title: string;
  children: ReactNode;
};

export function WwwPanel({ title, children }: WwwPanelProps) {
  return (
    <section className="rounded-lg border border-neutral-200 bg-white p-5">
      <h2 className="text-xl font-semibold text-neutral-950">{title}</h2>
      <div className="mt-3 text-sm text-neutral-700">{children}</div>
    </section>
  );
}
```

### Static HTML Fixtures

Plain `.html` files are allowed only for generated/static preview fixtures, export snapshots, and compatibility tests. They are not public route authoring surfaces for WWW starter apps. They must be normal HTML fragments or documents, and generated copies should live under `.dx/www/output` or another tool-owned fixture path rather than beside app-authored source. They must not use legacy wrapper tags, custom compiler-only component tags, or event syntax that a browser cannot parse directly.

### Icon Contract

Use `Icon` in TSX components and `<icon name="pack:play" />` only for DX-owned source syntax that is lowered by the toolchain. Runtime custom-element compatibility may use `<dx-icon>` when needed. Package workers should use the DX Icons lane and avoid adding standalone icon-package lanes.

### Forge Package Naming

Public package labels are DX-owned categories: Authentication, State Management, Data Fetching & Cache, Validation & Schemas, Database ORM, Backend Platform Client, Payments, Internationalization, Motion & Animation, Documentation System, Markdown & MDX Content, AI SDK, WebAssembly Bridge, and UI Components and Icons. Upstream names belong in provenance fields, not as the primary user-facing product names.

### Removed Public Syntax

The previous experimental page/component wrapper syntax is no longer a public framework surface. Do not add docs, templates, tests, or generated starters that require wrapper tags or compiler-only component tags. If an internal compiler lane still needs migration compatibility, keep that compatibility hidden behind explicit migration code and never present it as the current framework contract.
