# DX Forge Public Beta Quickstart

This guide is the shortest public beta path for a developer who wants to try DX Forge from a clean app folder, keep all evidence local, and avoid broad replacement claims. It starts with `dx forge init-app` for the simplest local path, or `dx forge beta-install` when a reviewed adoption-inclusive release bundle is already available, then moves through strict checks, the public quickstart route, CI artifacts, source-owned package review, and no `node_modules` proof.

Run commands from `G:\WWW`.

## What This Proves

- Forge can create a clean beta app from source-owned packages.
- The app includes package docs, receipts, a source manifest, a scorecard, and a strict check artifact.
- The quickstart route is static and generated by `dx prove vertical`.
- The review path keeps no `node_modules`.
- Public copy stays honest: this is not a universal npm replacement and does not claim full Next.js, npm, WordPress, or framework replacement.

## 1. Create The Beta App

Create the clean app and write the first evidence artifacts:

```powershell
cargo run --manifest-path .\www\Cargo.toml -p dx-www --bin dx-www -- forge init-app --project .\.dx\forge-beta-app --write --format markdown --output .\.dx\forge-beta-app\.dx\forge\init-app\init-app.md
```

The shorter installed-binary form is:

```powershell
dx forge init-app --project .\.dx\forge-beta-app --write
```

Expected evidence:

- `.dx\forge-beta-app\dx.config.toml`
- `.dx\forge-beta-app\pages\forge-adoption.html`
- `.dx\forge-beta-app\.dx\forge\source-manifest.json`
- `.dx\forge-beta-app\.dx\forge\receipts`
- `.dx\forge-beta-app\.dx\forge\docs`
- `.dx\forge-beta-app\.dx\forge\init-app\dx-check.json`
- `.dx\forge-beta-app\.dx\forge\init-app\forge-scorecard.json`

The app must not contain `.dx\forge-beta-app\node_modules`.

When a reviewed adoption-inclusive release bundle already exists, inspect it before install so the operator can see the signed manifest, hosted artifacts, rollback inputs, cache policy, package-gallery coverage, and no-`node_modules` boundary from the bundle alone:

```powershell
cargo run --manifest-path .\www\Cargo.toml -p dx-www --bin dx-www -- forge release-bundle-inspect --bundle .\.dx\forge-release-bundle-adoption --format markdown --output .\.dx\ci\forge-release-bundle-inspect.md --fail-under 90
```

Installed-binary form:

```powershell
dx forge release-bundle-inspect --bundle .\.dx\forge-release-bundle-adoption --format markdown --fail-under 90
```

The inspector should show `ready-for-beta-review` before the beta install path continues.

Use the bundle-based beta install to bootstrap the same clean app and write the full provenance/trust artifact trail in one step:

```powershell
cargo run --manifest-path .\www\Cargo.toml -p dx-www --bin dx-www -- forge beta-install --project .\.dx\forge-beta-app --release-bundle .\.dx\forge-release-bundle-adoption --artifacts .\.dx\forge-beta-app\.dx\forge\beta-install --write --format markdown --output .\.dx\forge-beta-app\.dx\forge\beta-install\beta-install.md --fail-under 90
```

Installed-binary form:

## 11. Secret And Scope Review

Before publishing the route or copy, run the automated copy gate:

```powershell
cargo run --manifest-path .\www\Cargo.toml -p dx-www --bin dx-www -- forge launch-copy-review --project . --copy .\docs\forge-public-beta-quickstart.md --copy .\docs\forge-public-launch-handoff.md --route-comparison .\benchmarks\reports\forge-public-route-comparison.json --source-review .\benchmarks\reports\forge-source-owned-package-review.json --static-evidence .\benchmarks\reports\forge-static-competitor-evidence.json --format markdown --output .\.dx\ci\forge-launch-copy-review.md --fail-under 90
```

Installed-binary form:

```powershell
dx forge launch-copy-review --project . --format markdown --fail-under 90
```

The report must block universal npm/framework replacement claims while preserving verified source-owned package security and static/no-runtime route-performance highlights.

Also check for blocked markers and overclaims:

- no `CLOUDFLARE_R2_`, `DX_FORGE_R2_LIVE`, `R2_SECRET`, or `SECRET_ACCESS_KEY` markers in public artifacts;
- no claim that Forge prevents every supply-chain attack;
- no claim that Forge replaces npm, Cargo, pip, WordPress, Next.js, or all frontend frameworks today;
- no claim of live customer adoption unless live customer evidence exists.
