## Public API

- `js/lib/automations/n8n/catalog.ts` maps the generated connector manifest shape with `display_name`, `credential_type_names`, resources, operations, and `workflow_node.usable_as_tool`.
- `js/lib/automations/n8n/readiness.ts` turns connector metadata into an app-owned credential readiness result and exposes `requiredEnvForDxN8nConnector` for connector-specific env gates.
- `js/lib/automations/n8n/receipt.ts` exposes `createDxN8nRunReceipt` for redacted DX/Zed run receipts in draft, dry-run, and run handoff modes, including the normalized operator workflow intent and required env boundary.
- `app/api/automations/n8n/dry-run/route.ts` exposes a source-owned App Router boundary for connector readiness and local dry-run receipts. It returns `runtimeExecution: false`, never reads provider secrets, and returns `missing-config` for live run handoffs when required env is absent.
- `examples/dashboard/src/lib/n8nAutomationBridge.ts` exposes `createN8nDashboardWorkflow`, `selectN8nConnector`, `buildN8nWorkflowReadiness`, `createRedactedN8nReceipt`, and `formatN8nCredentialBoundary`.

## Dashboard Usage

The generated `/launch` dashboard renders `LaunchAutomationBridgeStatus` inside `data-dx-component="launch-automation-dashboard-workflow"`. That surface lets an operator select Manual Trigger, Slack, or Notion metadata, inspect source files, credential schemas, auth kinds, credential type names, required env, trigger/tool readiness, write a release notification intent, and prepare a safe local redacted receipt without loading provider secrets. Its connector surface uses `data-dx-component="launch-automation-connector-workflow"`, `data-dx-component="launch-automation-catalog-summary"`, and `data-dx-automation-workflow="connector-readiness"` so Web Preview sees a product workflow instead of a placeholder card. The component's non-dashboard surface is named `connector-readiness`, and its local dry-run receipt uses that same workflow id rather than a preview/proof id. Local and Zed handoff receipt outputs now expose `data-dx-automation-receipt-intent`, `data-dx-automation-run-receipt-intent`, and `data-dx-automation-required-env`, backed by the typed `workflowIntent` and `requiredEnv` fields returned from `createDxN8nRunReceipt`. The workflow publishes `LaunchAutomationDashboardState` into `automation-mission-summary.tsx` so the mission dashboard summary at `data-dx-component="launch-automation-mission-summary"` updates the selected connector, workflow-node readiness, auth schema, credential types, required env, and `data-dx-automation-dashboard-state`. The starter dashboard also imports `AutomationWorkflowPanel` with `data-dx-package="automations/n8n"` and `data-dx-component="dashboard-automation-workflow"` for the same connector-readiness workflow.

DX Studio and Zed Web Preview discovery target the full `/launch` workflow section, not only the nested status widget. `dx www preview-manifest --json` exposes `data-dx-dashboard-workflow="automation-release-receipt"`, `data-dx-component="launch-automation-connector-workflow"`, `data-dx-component="launch-automation-catalog-summary"`, `data-dx-component="launch-automation-mission-summary"`, `data-dx-automation-workflow="connector-readiness"`, `data-dx-automation-dashboard-state`, `data-dx-automation-dashboard-card`, `data-dx-automation-intent-input`, `data-dx-automation-receipt-path`, `data-dx-automation-run-receipt-path`, `data-dx-automation-receipt-intent`, `data-dx-automation-run-receipt-intent`, `data-dx-automation-required-env`, `data-dx-automation-credential-schema`, `data-dx-automation-auth-kind`, `data-dx-automation-credential-type`, `data-dx-automation-workflow-node-readiness`, and `data-dx-automation-usable-as-tool` markers so source selection opens the dashboard workflow and redacted receipt boundary.

The visible workflow root, connector workflow, and mission summary also carry `data-dx-style-surface="automation-connectors"` and `data-dx-token-scope="automations/n8n"`. This is source-owned dx-style compatibility evidence for DX Studio and Zed, not browser visual proof or live n8n runtime proof.

`G:\Dx\.dx\receipts\automations\launch-release-notification.json` is now materialized as a redacted launch-dashboard receipt seed. It records the package id, workflow selector, connector auth schema, workflow-node readiness, required environment gates, and DX/Zed commands while keeping `runtime_execution` false and `secret_values` empty. This is a local draft handoff, not live n8n execution evidence.

`dx automations run --json` is aligned to that same launch workflow for Rust/Zed handoff. The `/launch` workflow now has a visible `data-dx-automation-safe-action="prepare-zed-run-handoff"` action and `data-dx-automation-handoff="zed-run-receipt"` output pointing at `G:\Dx\.dx\receipts\automations\run-latest.json`. Its redacted run receipt records `package_id: "automations/n8n"`, `dashboard_workflow: "automation-release-receipt"`, `source_selector: "[data-dx-component=\"launch-automation-dashboard-workflow\"]"`, the `launch-release-notification.json` receipt seed path, the redacted `workflow_intent` and `required_env` copied from that seed, `runtime_execution: false`, and an empty `secret_values` list before any real credentials or provider execution are allowed.
