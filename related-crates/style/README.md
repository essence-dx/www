# DX Style

DX Style is the Rust-owned CSS scanner and generator for DX-WWW. The public launch path is normal generated CSS: app files and style entry files are scanned, `styles/theme.dx.css` stays the semantic token source, and `styles/app.generated.css` is the CSS output that browsers load.

The launch pitch is intentionally narrow: Tailwind-familiar CSS workflow, Rust-powered scanner/generator, theme-token strictness, no hidden dependency surface, and PostCSS replacement for DX starters without a local PostCSS dependency chain. Full Tailwind classname parity remains gated by the compatibility matrix and guards.

Binary style output is internal and deprecated for the public launch path. The older DXC1/Dawn/binary modules still exist for experiments and internal compiler lanes, but the official DX-WWW launch story is generated CSS.

## Public Workflow

```bash
dx style build
dx style watch
dx style check
```

- `dx style build` writes `styles/app.generated.css` and a receipt under `.dx/receipts/style/`.
- `dx style watch` writes the same generated CSS plus a watch contract; it does not need PostCSS or a Node dependency install.
- `dx style check` verifies required theme tokens, generated CSS freshness, unused generated classes, hardcoded color leakage, and Tailwind dependency leakage.

## What It Scans

DX-WWW's public command path scans project-owned app and component files plus style entry files:

- `app/**/*.tsx`
- `app/**/*.jsx`
- `app/**/*.mdx`
- `app/**/*.html`
- `components/**/*.tsx`
- `components/**/*.jsx`
- `components/**/*.mdx`
- `components/**/*.html`
- `forge/**/*.tsx`
- `forge/**/*.jsx`
- `forge/**/*.mdx`
- `forge/**/*.html`
- `styles/**/*.css`
- `styles/theme.dx.css`
- `styles/globals.css`
- `forge/**/*`

Only `.tsx`, `.jsx`, `.ts`, `.js`, `.html`, and `.mdx` files feed static class-token extraction. CSS files remain style entry inputs, so authored `styles/theme.dx.css` and `styles/globals.css` can participate in the normal generated CSS path without being misreported as scanned class sources.

The public `dx style build` and `dx style check` receipts expose this boundary directly through `class_source_extension_counts` and `style_entry_file_count`, giving DX-WWW, Forge, Check, Zed, and Friday a stable read model instead of requiring each consumer to re-walk source paths.

The style crate itself also has parsers for HTML/JSX class attributes, grouping syntax, arbitrary values, states, screens, container queries, a CSS registry, cache, serializer integration, watcher modules, and theme/color systems.

## Tailwind Compatibility

DX Style supports a useful Tailwind-familiar subset, not every Tailwind class. The maintained status table is in `TAILWIND_COMPATIBILITY.md`.

The Tailwind v4.3 compatibility matrix is now source-owned in `src/core/engine/feature_matrix.rs` and mirrored in `TAILWIND_COMPATIBILITY.md`. The latest checked target for this matrix is `tailwindcss@4.3.0`. dx-style does not depend on Tailwind, does not execute Tailwind JS config/plugins, and treats drop-in Tailwind replacement behavior as matrix-gated compatibility work.

CSS-first compatibility is DX-owned, not a hidden Tailwind runtime. `@import "tailwindcss"` is accepted only as migration input and stripped/replaced by dx-style; `@import "tailwindcss" source(none)` is converted into DX-owned `@source none`, disabling automatic app/component scanning while preserving explicit `@source` roots and inline safelists. `@theme` tokens feed the Rust token engine, local CSS `@reference "./tokens.css"` inputs can share theme tokens in the public build path, `@reference "tailwindcss"` is consumed as a DX-owned default-theme reference without loading the Tailwind package, static `@source` paths, `@source none`, and `@source not` exclusions shape scanning, `@source inline(...)` safelists expand brace options and numeric ranges, `@source not inline(...)` subtracts explicit class families from generation, safe `@utility` declarations, one trailing `*` functional segment, and one-level top-level nested selectors beginning with `&` can register utilities, functional `@utility` values support common theme-key/bare/arbitrary/literal `--value(...)`, `--modifier(...)`, v4.3 `--default(...)`, and `@utility`-local `--alpha()`/`--spacing()` helpers, standalone authored CSS declarations can lower safe `--alpha()` and `--spacing()` calls, safe `@custom-variant` shorthand selector wrappers and block-form `@slot` selector/media wrappers register variants, and authored CSS supports common `@variant` pseudo/dark wrappers plus v4.3 stacked and compound forms such as `@variant hover:focus` and `@variant hover, focus`. Unsupported Tailwind JS surfaces such as `@plugin`, `@config`, and legacy `@tailwind` directives are reported in receipts instead of silently ignored.

Source scanner coverage now includes complete static quoted class strings in Tailwind-style object maps, arbitrary static string candidates, and explicit rejection canaries for dynamic interpolation such as `bg-${color}-600`. This is source-owned plain-text scanning coverage, not full Tailwind source graph or prose-token filtering parity.

DX-native grouped classnames are scanner syntax, not hidden CSS magic: `hover:(bg-accent text-accent-foreground shadow-sm)`, `md:(grid grid-cols-2 gap-4)`, `dark:hover:(bg-card text-foreground)`, and `group-hover:(opacity-100 translate-y-0)` expand into normal generated class tokens before CSS generation. Parenthesized token utilities such as `bg-token(surface)` and `bg-size-(--dx-bg-size)` stay intact. Invalid unclosed groups emit an explicit `dx-grouping-error:*` scanned token so `dx style check` can report the bad syntax instead of silently compiling the wrong CSS.

## Cache And Binary Evidence

- `memmap2` is a dx-style dependency in `Cargo.toml`.
- `src/cache/snapshot.rs` reads cache snapshots with `memmap2::Mmap`.
- `src/core/engine/mod.rs` maps `.dx/style/style.dxm` with `memmap2::{Mmap, MmapOptions}`.
- `src/core/output.rs` uses `memmap2::MmapMut` for output backends.
- `rkyv is not a dx-style dependency` in this crate today. It appears elsewhere in the workspace lockfile, but dx-style does not currently import or depend on it.

## Registry snapshot plan

`src/cache/registry_snapshot_plan.rs` now defines the isolated cached registry/theme-token snapshot contract for the next performance pass. The plan is intentionally small: `.dx/style/registry.snapshot`, a format version, class registry keys, theme token names/resolved values, screen/container token names, and cache checksums. It is designed as a future memmap2-backed read-only memory map after atomic writes.

rkyv remains unwired in dx-style. This plan does not add a dependency, does not change runtime I/O, and does not move binary style output into the public launch pitch. It can become active only after a focused Rust compile/CLI smoke pass is allowed.

## Quality Bar

Keep the launch story evidence-backed:

- Add guards for every newly claimed supported class family.
- Update `TAILWIND_COMPATIBILITY.md` when support moves from partial to supported.
- Keep public docs centered on normal generated CSS.
- Keep PostCSS/Tailwind dependency leakage out of official starters.
- Do not market binary style output until it has a governed public path again.
